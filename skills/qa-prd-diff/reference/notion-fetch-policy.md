# Notion PRD 수집 정책 (notion-fetch-policy)

> `qa:prd-clarify` · `qa:generate-tc` · `qa:prd-diff`가 Notion PRD를 fetch할 때 공통 적용.
> 목적: "메인 페이지만 읽었는데 핵심 스펙이 하위 페이지에 있었다"류 누락 방지 + 무엇을 읽었는지 투명화.

## 1. 도구 감지

Notion 도구명은 환경마다 다르다 (`notion-fetch`, `mcp__claude_ai_Notion__fetch`, `mcp__notion__*` 등).

- 도구 목록에서 이름에 `notion`이 포함된 fetch/search 계열 도구를 런타임에 감지해 사용.
- 감지 실패 시: "Notion MCP가 연결 안 됨. PRD 본문을 직접 붙여줄래?" 안내 후 중단. 도구명을 추측해 하드코딩하지 않는다.

## 2. 수집 범위 (1-depth 규칙)

| 소스 | 처리 |
|---|---|
| 메인 페이지 본문 + 자식 블록 | 필수 fetch |
| 본문 내 **내부 Notion 링크** (하위 페이지·관련 문서) | **1-depth follow 필수** — 링크된 페이지 본문을 fetch해서 분석에 포함. 그 페이지 안의 링크(2-depth)는 follow 안 함 |
| 인라인 데이터베이스 | fetch해서 행 목록 포함. 행 50개 초과 시 처음 50행 + "이하 생략" 명시 |
| 외부 링크 (Google Docs·Figma·Jira 등) | 자동 follow 안 함. 발견 사실을 소스 매니페스트에 기록하고, 스펙 핵심으로 보이면 사용자에게 포함 여부 확인 |
| 첨부 PDF | Notion fetch로 본문 추출 불가 — 사용자에게 로컬 경로(Code) 또는 업로드(Desktop) 요청. 받으면 직접 분석 |
| 임베드 이미지·다이어그램 | fetch 결과에 텍스트로 안 들어옴 — §3 참조 |

내부 링크가 10개를 넘으면 전부 follow하지 말고, 링크 목록을 보여주고 어떤 것을 포함할지 사용자에게 묻는다.

## 3. 이미지·스크린샷 입력

- 사용자가 직접 준 스크린샷(PRD 캡처·Figma 화면·정책표 이미지)은 **직접 분석 가능** — Code: 로컬 이미지 경로를 받아 읽기 / Desktop: 업로드.
- Figma는 MCP 연결 시 프레임 구조·실제 텍스트·상태별 화면까지 읽을 수 있어 스크린샷보다 우선. 미연결이면 스크린샷으로 대체 가능 — 단, 숨은 상태(빈/에러/로딩)는 스크린샷에 없을 수 있음을 리포트에 명시.
- Notion 페이지 안 임베드 이미지는 fetch가 못 읽는다. 플로우차트·정책표가 이미지로만 존재한다고 추정되면: "이 위치의 이미지가 스펙 핵심으로 보임 — 스크린샷 붙여줄래?" 요청.

## 4. 소스 매니페스트 (의무)

분석 산출물(클래리파이 리포트·TC 미리보기·diff 리포트) **머리에** 무엇을 읽었고 무엇을 못 읽었는지 명시:

```markdown
> **분석 소스**
> - ✅ 메인 PRD: <제목> (<URL>)
> - ✅ 하위 페이지: <제목> (1-depth)
> - ⏭️ 외부 링크 미포함: <Google Docs 링크> — 필요 시 요청
> - ⚠️ 임베드 이미지 2개 분석 불가 (플로우차트 추정) — 스크린샷 제공 시 보강 가능
```

추가 소스가 하나도 없어도 "메인 페이지만 분석" 한 줄은 항상 남긴다.
