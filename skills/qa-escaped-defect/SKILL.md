# qa:escaped-defect

QA를 통과해 프로덕션으로 나간 버그를 역추적한다. "어느 단계에서 신호를 놓쳤는가"를 판정하고, 같은 유형이 다시 새지 않도록 보강 TC 후보를 만든다.

## 원칙: Blameless

이 스킬은 "누가 실수했나"가 아니라 "프로세스의 어느 단계에서 신호를 놓쳤는가"를 찾는 도구다.
- 갭 A~D는 전부 개선 기회다. 개인 귀속 표현(담당자 이름·"~가 놓침")을 리포트에 쓰지 않는다.
- 리포트의 목적은 다음 라운드를 더 잘하기 위한 정보이지 책임 추궁이 아니다.

## 입력
- **필수** (없으면 추정하지 말고 질문):
  - 버그 증상·실제 동작
  - 발생 환경 (기기·OS·앱 버전 — 대략이라도)
  - 영역 또는 재현 단계 중 하나
  - 당시 실행한 TC xlsx + 탭
- **옵션**: Jira 이슈 키 (Atlassian MCP 연결 시 fetch — 미연결이면 텍스트로 요청), PRD/스냅샷 (스펙 대조 정확도 상승)

## 워크플로우

### 1. 버그 정보 수집
- Jira 키를 받았고 Atlassian MCP가 연결돼 있으면 이슈를 fetch해 증상·환경·재현 단계를 채운다. 미연결이면 본문 붙여넣기 요청.
- 필수 항목이 비면 질문하고 중단. 추정으로 채우지 않는다.

### 2. 관련 TC 추적
```bash
uv run scripts/extract_tc_table.py <xlsx> --tab <탭>
```
평탄화된 행에서 버그의 영역·키워드(`reference/risk-taxonomy.md` 영역 분류 참조)와 관련된 TC를 찾는다. 관련 TC마다: TC_ID / Test Summary / 당시 Result / 버그 시나리오와의 거리(조건 일치 여부).

### 3. 갭 유형 판정 (정확히 하나, 근거 필수)
- **A. 미작성** — 버그 시나리오를 커버하는 TC 자체가 없음
- **B. 미실행** — 커버 TC가 있으나 당시 Result가 N/T·미입력
- **C. 실행했으나 못 잡음** — 커버 TC가 Pass였으나 스텝·조건이 버그 발생 조건을 안 덮음 (**빠진 조건을 구체적으로 명시** — 예: "백그라운드 전환 타이밍 없음")
- **D. Fail이었는데 릴리즈** — 당시 Fail/Block이었음에도 릴리즈됨 → 프로세스 갭. `qa:release-checklist` 게이트(G1·G2) 운영 점검 권고

판정 근거가 부족하면 단정하지 말고 후보 유형 2개와 확인 질문을 제시한다.

### 4. 보강 TC 후보 (갭 A·C일 때)
모호점 핸드오프 규약 형식의 마크다운 표 — `qa:generate-tc`에 바로 입력 가능:
- 기대 결과를 확정할 수 있으면 Expected Result에 명시
- 확정 불가면 Expected Result `TBD (PM 확인 필요)` + Comment 첫 줄 `(Blocker) <질문 요약>`
- 버그 발생 조건(환경·타이밍·상태)을 Pre-condition·Test Step에 반드시 포함

### 5. 출력 (md)
```markdown
## 유출 결함 분석 — <버그 한 줄 요약>
### 버그 요약
(증상 / 환경 / 영역 — 출처: 텍스트 or Jira <키>)
### 관련 TC 추적
| TC_ID | Test Summary | 당시 Result | 버그 시나리오와의 차이 |
### 갭 유형: <A~D> — <이름>
(근거 — 어느 단계에서 신호를 놓쳤는지)
### 보강 TC 후보
(핸드오프 규약 표 — A·C일 때)
### 프로세스 제안
(D면 release-checklist 게이트 점검, B면 회귀 범위 규칙(`qa:regression-scope`) 보강 등 — 1~3개만)
```

### 6. 다음 액션 안내
"보강 TC 후보는 `qa:generate-tc`에 이 리포트를 입력으로 넘기면 핸드오프 규약으로 반영됩니다."

## 비목표
- 버그 원인 코드 분석·수정
- TC 자동 추가 (후보 제안만 — 추가 여부는 사람이 결정)
- 책임 귀속·개인 평가
