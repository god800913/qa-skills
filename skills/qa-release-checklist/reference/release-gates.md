# 릴리즈 게이트 정의

릴리즈 판정 시 반드시 점검하는 7개 게이트. 각 게이트의 통과 기준과 확인 불가 처리 방침을 정의한다.

> **원칙**: 확인 불가 게이트가 1개라도 있으면 `ready` 판정 불가 — `conditional`이 상한.

---

## 게이트 정의표

| # | 게이트 | 통과 기준 | 확인 불가 시 처리 |
|---|--------|-----------|------------------|
| G1 | **Blocker 결함 0건** | 릴리즈 범위 내 Blocker 등급 결함이 전부 Close/Verified 상태. 미결 Blocker가 1건이라도 있으면 즉시 실패. | 정보 없음 → `확인 불가` 표시. 판정 상한 `blocked` (Blocker 가능성 배제 불가). |
| G2 | **Major 결함 해결 또는 risk-accept 서명** | (a) 미결 Major 0건, 또는 (b) 잔여 Major 전건에 PM·개발 책임자의 risk-accept 서명(날짜·성명·사유 포함) 존재. | 정보 없음 → `확인 불가` 표시. 판정 상한 `conditional`. |
| G3 | **known issue 문서화 + CS 공유** | 릴리즈와 함께 출시되는 known issue 전건이 문서화되어 있고, CS팀에 공유 완료(날짜 확인 가능). known issue 없음도 명시적 확인 필요. | 정보 없음 → `확인 불가` 표시. 판정 상한 `conditional`. |
| G4 | **rollout 계획 수립** | 단계 배포 비율(예: 5% → 20% → 100%)과 각 단계 Remote Config 기본값이 문서화. 전체 배포 시에도 계획 문서 존재. | 정보 없음 → `확인 불가` 표시. 판정 상한 `conditional`. |
| G5 | **rollback 절차 확보** | 롤백 트리거 조건(예: 크래시율 임계값, 오류율 임계값)과 담당자(개발·인프라 연락처)가 명시된 절차서 존재. | 정보 없음 → `확인 불가` 표시. 판정 상한 `conditional`. |
| G6 | **모니터링 준비** | 릴리즈 직후 관찰할 핵심 지표(예: 결제 성공률, 크래시율, API p99 응답시간)와 알람 임계값이 설정 완료. | 정보 없음 → `확인 불가` 표시. 판정 상한 `conditional`. |
| G7 | **owner sign-off** | QA 리드·PM·개발 책임자 3자의 릴리즈 승인(서면 또는 메신저 캡처)이 모두 확보. 1인이라도 미확인 시 실패. | 정보 없음 → `확인 불가` 표시. 판정 상한 `conditional`. |

---

## 판정 규칙 요약

| 판정 | 조건 |
|------|------|
| `blocked` | G1 실패, 또는 G1 확인 불가 |
| `conditional` | G1 통과 + G2~G7 중 1개 이상 실패/확인 불가 — 잔여 조건 명시 필수 |
| `ready` | G1~G7 전 게이트 통과 (또는 합리적 예외가 근거와 함께 기록됨) |

> **운용 지침**: 확인 불가 게이트는 "해당 없음"이 아니다. 해당 없음으로 처리하려면 입력에 명시적 근거가 있어야 한다.
